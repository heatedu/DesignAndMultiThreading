================================================================================
SOLID PRINCIPLES DEMONSTRATED
================================================================================

‚úÖ S - Single Responsibility
   - Vehicle: manages vehicle data only
   - Reservation: manages booking data only
   - Store: manages inventory only
   - CarRentalSystem: coordinates everything

‚úÖ O - Open/Closed
   - Can add new vehicle types without modifying existing code
   - Just inherit from Vehicle and override calculatePrice()

‚úÖ L - Liskov Substitution
   - Any Vehicle* can point to Economy/Luxury/SUV
   - All work correctly in place of base class

‚úÖ I - Interface Segregation
   - Small, focused interfaces
   - No fat interfaces forcing unused methods

‚úÖ D - Dependency Inversion
   - System depends on Vehicle abstraction
   - Not on concrete Economy/Luxury/SUV classes

================================================================================
KEY ALGORITHMS IMPLEMENTED
================================================================================

1. CONFLICT DETECTION (Reservation.h)
   bool hasConflict(time_t start, time_t end) {
       if (cancelled) return false;
       return !(end <= startTime || start >= endTime);
   }
   
   Logic: Two time ranges overlap if:
   - NOT (one ends before other starts OR one starts after other ends)

2. AVAILABILITY SEARCH (CarRentalSystem.h)
   - Get all vehicles at location
   - Filter by type
   - Check each against existing reservations
   - Return only conflict-free vehicles

3. POLYMORPHIC PRICING (Vehicle.h)
   - Base class declares pure virtual calculatePrice()
   - Each subclass implements specific pricing logic
   - Runtime polymorphism via virtual function table
